# 记忆压缩机制研究框架

## 📋 研究背景

### 问题提出

**Human提出**：
> 记忆压缩具体实现模式具有难度，如何压缩且保留关键信息。

**背景**：
- 理论设计了记忆压缩机制
- 会话结束：session-log.md → CATCH_UP.md（摘要）
- 项目里程碑：CATCH_UP.md → experiences/（经验提取）
- 但具体实现方式未明确

**核心挑战**：
1. 如何压缩？（压缩算法）
2. 保留什么？（关键信息识别）
3. 何时压缩？（触发条件）
4. 压缩效果？（评估方法）

---

## 🎯 研究目标

### 主要目标

**目标1：设计记忆压缩算法**
- 研究如何从大量文本中提取关键信息
- 设计适合Agent记忆的压缩策略
- 平衡信息完整性和压缩率

**目标2：定义关键信息保留规则**
- 明确什么是"关键信息"
- 设计关键信息识别机制
- 确保关键信息不丢失

**目标3：确定压缩触发条件**
- 定义何时需要压缩
- 设计自动化触发机制
- 避免过早或过晚压缩

**目标4：建立压缩效果评估方法**
- 设计压缩质量指标
- 建立评估流程
- 支持持续优化

---

## 🔍 问题分析

### 问题1：压缩什么？

**记忆层次**：
```
会话记忆（session-log.md）
  - 内容：详细的会话过程
  - 特点：短期、临时、详细
  - 大小：~2k tokens/会话
  - 问题：信息冗余大

状态记忆（CATCH_UP.md）
  - 内容：当前状态、最近历史
  - 特点：中期、项目级
  - 大小：~3k tokens
  - 问题：会累积历史信息

经验记忆（experiences/）
  - 内容：历史经验、知识积累
  - 特点：长期、按主题组织
  - 大小：可变
  - 问题：可能过度积累
```

**压缩场景**：
```
场景1：session-log → CATCH_UP
  触发：会话结束时
  目标：提取关键决策和任务
  挑战：如何识别"关键"？

场景2：CATCH_UP → experiences
  触发：项目里程碑时
  目标：提取经验教训
  挑战：如何从状态中提取经验？

场景3：experiences → 知识库
  触发：长期未使用时
  目标：归档冷知识
  挑战：如何判断"冷知识"？
```

---

### 问题2：如何识别关键信息？

**关键信息定义**：
```
关键信息 = 决策 + 问题 + 结果 + 经验

决策信息：
  - 重要决策
  - 决策理由
  - 决策影响

问题信息：
  - 遇到的问题
  - 解决方案
  - 是否有效

结果信息：
  - 任务结果
  - 验收情况
  - 质量评估

经验信息：
  - 经验教训
  - 最佳实践
  - 改进建议
```

**识别难点**：
```
难点1：什么是"重要"？
  - 对谁重要？
  - 重要程度如何量化？
  - 主观性如何处理？

难点2：如何自动识别？
  - 需要NLP技术？
  - 需要规则引擎？
  - 需要人工标注？

难点3：上下文如何保留？
  - 关键信息需要上下文
  - 压缩可能丢失上下文
  - 如何平衡？
```

### 问题3：压缩算法选择

**可选方案**：

#### 方案A：摘要式压缩
```
方法：
  - 使用LLM生成摘要
  - 保留关键句子
  - 删除冗余信息

优点：
  ✅ 简单直接
  ✅ 保留语义
  ✅ 可读性好

缺点：
  ❌ 需要LLM调用
  ❌ 成本较高
  ❌ 可能遗漏细节
```

#### 方案B：提取式压缩
```
方法：
  - 提取关键句子/段落
  - 按规则筛选
  - 组合成摘要

优点：
  ✅ 不需要LLM
  ✅ 速度快
  ✅ 保留原文

缺点：
  ❌ 可能不连贯
  ❌ 规则难设计
  ❌ 缺少语义理解
```

#### 方案C：结构化压缩
```
方法：
  - 解析文档结构
  - 提取结构化信息
  - 存储为YAML/JSON

优点：
  ✅ 机器可读
  ✅ 信息完整
  ✅ 易于检索

缺点：
  ❌ 需要解析逻辑
  ❌ 可能丢失语义
  ❌ 可读性差
```

#### 方案D：分层压缩
```
方法：
  - 分层压缩（高/中/低）
  - 不同层保留不同信息
  - 按需加载不同层

优点：
  ✅ 灵活
  ✅ 可控
  ✅ 支持按需加载

缺点：
  ❌ 复杂度高
  ❌ 需要分层规则
  ❌ 管理成本高
```

### 问题4：触发条件设计

**触发时机**：
```
时机1：时间触发
  - 会话结束：N分钟后
  - 定期压缩：每天/每周
  - 时机问题：如何定义"会话结束"？

时机2：大小触发
  - 文件大小超过阈值
  - Token数量超过阈值
  - 问题：阈值如何设定？

时机3：事件触发
  - 任务完成
  - 里程碑达成
  - 问题：事件如何检测？

时机4：主动触发
  - Agent主动判断需要压缩
  - Human手动触发
  - 问题：如何自动化？
```

## 💡 初步研究思路

### 思路1：基于结构化的压缩

**核心思想**：
```
不使用LLM摘要，而是：
1. 解析文档结构（Markdown结构）
2. 提取结构化字段（决策、问题、结果）
3. 转换为YAML/JSON格式
4. 保留关键信息模板
```

**示例**：
```markdown
# 原始session-log.md

## [2026-03-07 10:15] 会话开始

### 任务
- 响应 Research Agent 的观察要求
- 准备启动 v1.1 Sprint 1

### 关键决策
- 接受 Research Agent 的观察要求
- 按照指定格式创建日志

### 遇到的问题
- 无

### 与其他Agent的交互
- 接收到 Research Agent 的观察要求文档
```

**压缩后（YAML）**：
```yaml
session:
  date: "2026-03-07"
  start_time: "10:15"
  
tasks:
  - "响应 Research Agent 的观察要求"
  - "准备启动 v1.1 Sprint 1"

decisions:
  - description: "接受 Research Agent 的观察要求"
    impact: "创建 session-log.md"
  
problems: []

interactions:
  - with: "Research Agent"
    type: "接收观察要求"
```

**优势**：
- ✅ 不需要LLM
- ✅ 结构化、机器可读
- ✅ 信息完整保留
- ✅ 易于检索

**劣势**：
- ⚠️ 需要解析规则
- ⚠️ 格式依赖模板
- ⚠️ 灵活性受限

---

### 思路2：基于模板的压缩

**核心思想**：
```
定义压缩模板：
1. 为每种记忆定义模板
2. 从原文中提取字段
3. 填充到模板
4. 生成压缩后文档
```

**模板示例**：
```yaml
# CATCH_UP压缩模板

template: "catch-up-compression"

required_fields:
  - last_updated: "最后更新时间"
  - current_phase: "当前阶段"
  - completed_tasks: ["已完成任务列表"]
  - next_actions: ["下一步行动"]

optional_fields:
  - blockers: ["阻塞问题"]
  - risks: ["风险"]
  - notes: ["备注"]

compression_ratio: "目标压缩率30-50%"
```

**执行流程**：
```
1. 读取CATCH_UP.md
2. 解析字段：
   - 提取"Last Updated"
   - 提取"Current Phase"
   - 提取"Completed Tasks"列表
   - ...
3. 填充模板
4. 生成压缩后的CATCH_UP.md
5. 备份原文件
```

**优势**：
- ✅ 可控性强
- ✅ 信息完整
- ✅ 易于实施

**劣势**：
- ⚠️ 模板需要预先设计
- ⚠️ 不够灵活
- ⚠️ 需要维护模板

---

### 思路3：基于重要性的分层压缩

**核心思想**：
```
分层压缩：
Layer 0: 完整信息（原文件）
Layer 1: 重要信息（摘要）
Layer 2: 关键信息（要点）
Layer 3: 元信息（索引）

按需加载不同层
```

**示例**：
```
session-log.md（Layer 0，~2k tokens）
  ↓ 压缩
session-summary.md（Layer 1，~500 tokens）
  - 关键决策摘要
  - 主要任务列表
  ↓ 压缩
session-keypoints.md（Layer 2，~100 tokens）
  - 决策要点：3条
  - 任务要点：5条
  ↓ 压缩
session-index.yaml（Layer 3，~20 tokens）
  - 日期：2026-03-07
  - 决策数：3
  - 任务数：5
  - 关键词：[Sprint, Research, Issue]
```

**加载策略**：
```
启动时：
  - 加载Layer 3（索引）
  - 快速了解历史

需要详情时：
  - 加载Layer 2（要点）
  - 了解关键信息

深入研究时：
  - 加载Layer 1（摘要）
  - 获取完整上下文

需要原文件时：
  - 加载Layer 0（完整）
  - 查看所有细节
```

**优势**：
- ✅ 灵活加载
- ✅ 平衡完整性和效率
- ✅ 支持渐进式加载

**劣势**：
- ⚠️ 实施复杂
- ⚠️ 存储空间增加
- ⚠️ 管理成本高

## ❓ 需要讨论的问题

### 问题1：压缩目标

**讨论点**：
- 压缩率目标是多少？（50%？70%？）
- 信息保留率目标是多少？（80%？90%？）
- 是否接受信息损失？

**我的初步想法**：
```
目标压缩率：50-70%
目标信息保留率：>85%
接受一定程度的细节损失，但关键信息必须保留
```

---

### 问题2：压缩是否需要LLM

**讨论点**：
- 是否可以使用纯规则压缩？
- 还是必须使用LLM摘要？
- 成本 vs 效果如何平衡？

**我的初步想法**：
```
优先考虑规则压缩（基于结构化、模板）
减少LLM依赖，降低成本
只在复杂场景考虑LLM辅助
```

---

### 问题3：分层压缩的必要性

**讨论点**：
- 是否需要分层压缩（Layer 0-3）？
- 还是单层压缩即可？
- 复杂度 vs 收益如何权衡？

**我的初步想法**：
```
先实现简单的单层压缩
观察效果后再决定是否需要分层
避免过度设计
```

---

### 问题4：人工干预的程度

**讨论点**：
- 压缩是否需要人工审核？
- 还是完全自动化？
- 如何平衡自动化和质量？

**我的初步想法**：
```
Phase 1：人工审核
  - Agent压缩后，Human审核
  - 确保质量

Phase 2：自动化 + 抽检
  - 自动压缩
  - Human抽检
  - 持续优化

Phase 3：完全自动化
  - 规则成熟后
  - 完全自动化
```
